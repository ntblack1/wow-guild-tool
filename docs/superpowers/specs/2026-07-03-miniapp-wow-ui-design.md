# 微信小程序魔兽风 UI 与 Q 版配图设计

## 背景

`wow-guild-tool` 当前阶段重点是“成员-角色-活动-报名-签到”的第一条业务闭环。此次设计只优化微信小程序前端视觉和配图，不改后端接口、数据库模型、登录、支付、复杂权限、排行榜或活动海报。

用户确认的视觉方向是：高幻想工会活动工具、深色金边面板、羊皮纸/皮革质感、可爱的 Q 版职业角色、副本活动卡、角色名片，以及清晰的报名和签到状态。

## 范围

采用 B 路线：核心页面视觉重做 + 关键配图。

本次优先覆盖：

- 首页 `apps/miniapp/pages/index`
- 活动详情页 `apps/miniapp/pages/event-detail`
- 报名页 `apps/miniapp/pages/signup`
- 我的角色页 `apps/miniapp/pages/my`
- 全局主题 `apps/miniapp/app.wxss` 与 `apps/miniapp/app.json`
- 小程序静态图片资源目录

暂不覆盖：

- 团长页完整重排
- 调试页视觉优化
- 微信登录
- 支付
- 复杂权限
- 排行榜
- 活动海报
- 后端接口或 Prisma schema 修改

## 视觉原则

整体界面应像“八块腹肌工会活动大厅”，不是普通后台表单。

视觉关键词：

- 深色背景：接近地下城大厅、战役面板，而非纯黑。
- 金色边框：用于卡片、主按钮、状态强调，但要避免过度发光。
- 羊皮纸/皮革质感：用轻量背景色、边框和阴影表达，不依赖复杂 CSS。
- 翡翠绿：用于报名中、成功、确认类状态。
- 暗红/铜色：用于取消、警告、危险操作。
- 职业色标签：坦克、治疗、近战、远程使用稳定可识别颜色。
- Q 版角色：可爱、圆润、原创幻想职业，不使用官方 logo 或现成角色形象。

信息优先级必须高于装饰。移动端文字要清楚，按钮要易点，列表要可扫读。

## 页面设计

### 首页

首页定位为活动大厅。

结构：

- 顶部工会欢迎区：展示“八块腹肌活动大厅”、后端状态、刷新入口。
- 活动列表标题区：展示近期活动，保留刷新能力。
- 活动卡片：每条活动显示副本图、活动标题、副本名、状态徽章、开始时间、人数上限或报名进度。

卡片设计：

- 左侧或顶部使用副本氛围图。
- 右侧/下方展示活动信息。
- 状态徽章使用颜色区分：报名中、已锁定、已结束、未开始等。
- 卡片点击仍进入活动详情。

### 活动详情页

活动详情定位为副本战报页。

结构：

- 顶部副本英雄区：副本图、活动标题、副本名、状态徽章、时间。
- 关键信息面板：团长、人数上限、职业需求、活动说明。
- 报名统计面板：已报名、候补、请假、取消，坦克/治疗/近战/远程需求和缺口。
- 操作区：报名按钮与查看报名名单按钮。
- 报名名单：保持清晰列表，使用角色和职责标签。

逻辑：

- 不改变现有 `loadDetail`、`goSignup`、`toggleSignups`、`loadSignups` 行为。
- 可以新增前端展示字段，例如角色标签 class、进度百分比、视觉状态 class。

### 报名页

报名页定位为选择出战角色。

结构：

- 顶部说明卡：展示当前成员与报名意图。
- 成员未选择/无角色状态：保留原有提示，但换成主题化空状态。
- 表单卡：选择角色、选择职责、备注。
- 主按钮：文案保留提交报名，视觉改成确认出战按钮风格。

逻辑：

- 不改变提交行为和校验逻辑。
- 表单可用性优先，输入框和 picker 必须保持明显可点。

### 我的角色页

我的角色页定位为角色名册。

结构：

- 成员信息卡：当前成员、配置兜底、刷新。
- 成员切换卡：保留 picker 和清除按钮。
- 角色列表：每个角色显示 Q 版头像或职业头像、角色名、主号标识、职业、专精、职责、装等。
- 角色表单：保持创建/编辑能力，视觉主题化。
- 报名记录：活动卡片化展示。

逻辑：

- 不改变成员切换、角色新增、编辑、设为主号、删除行为。
- 角色头像可以先使用静态职业/职责配图映射，不引入上传头像。

## 配图策略

先生成并落地 4-6 张原创图片：

- 工会大厅/活动首页背景或顶部插画
- 副本活动图 2-3 张
- Q 版坦克角色
- Q 版治疗角色
- Q 版输出角色，可包含法系或猎人风格

图片要求：

- 原创高幻想风格，不使用任何真实游戏 logo、官方徽记或现成角色。
- 可爱 Q 版，适合小程序移动端。
- 色调与深色金边 UI 统一。
- 图片文件放入小程序静态资源目录，例如 `apps/miniapp/assets/images/`。
- 若最终页面引用图片，不能只保留在 Codex 生成目录。

## 样式系统

全局新增或调整通用类：

- 页面背景与基础文字颜色
- 游戏面板卡片
- 金边标题
- 状态徽章
- 职责标签
- 人数/进度统计块
- 主题按钮、次级按钮、危险按钮
- 主题输入框、textarea、picker
- 空状态

需要控制样式复杂度。微信小程序端避免依赖复杂浏览器 CSS 特性，尽量使用稳定的 WXSS 属性。

## 数据与接口

不改后端接口，不改 Prisma schema，不新增 migration。

前端可以基于已有字段做展示增强：

- `event.status`
- `event.raidName`
- `event.startTime`
- `event.maxPlayers`
- `summary.signedCount`
- `summary.tankSigned`
- `summary.healerSigned`
- `summary.meleeSigned`
- `summary.rangedSigned`
- `character.roleType`
- `character.className`
- `character.spec`
- `character.itemLevel`

## 测试与验收

完成后至少运行：

```bash
npm run test
npm run build
```

验收标准：

- 首页、活动详情、报名、我的角色页统一呈现高幻想工会风。
- Q 版配图已经保存到项目静态资源目录并被页面引用。
- 页面可读性不低于原始版本。
- 关键按钮和表单控件在移动端尺寸足够。
- 不改变现有业务行为和接口调用。
- 测试与构建通过；如本地环境导致无法验证，需记录原因。

## 风险

- AI 生成图片可能出现文字错误或风格不一致，因此页面中不依赖图片内文字。
- 深色游戏 UI 容易牺牲可读性，需要控制对比度。
- 配图体积可能影响小程序加载，需要控制图片数量和尺寸。
- 当前部分源文件在终端显示中文乱码，编辑时应避免无关文案重写，尽量只改必要结构与样式。
